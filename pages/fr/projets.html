Objectif : Modéliser et visualiser l'exposition des petites et moyennes entreprises (PME) à la volatilité des prix de l'électricité en Europe. Connecter les prix historiques spot de l'électricité (données OPSD) à un modèle financier simplifié de P&L par segment de PME.

Stack :

  • Cloud & Infrastructure : GCP (Google Cloud Storage, BigQuery), Terraform (IaC)
  • Transformations & Modélisation : dbt Core (BigQuery, partitionnement mensuel, clustering, staging + mart, seeds)
  • Ingestion & Automatisation : Python (avec uv), tables externes BigQuery, scripting Shell
  • Serving / Dashboard : Looker Studio

Architecture & Optimisations BI

Flux d'Ingestion : Script Python (gestionnaire uv) -> Chargement du CSV OPSD brut sur GCS -> Table externe BigQuery (`raw_energy_prices_ext`).
Pipeline d'Analytique (dbt) : Transformation & Unpivot (staging) -> Cross Join dynamique avec les profils de PME (seeds) -> Table finale `mart_sme_energy_costs`.
Optimisation FinOps : Partitionnement mensuel sur `ts_utc` et clustering par `price_zone` et `segment` dans BigQuery pour optimiser le temps de réponse et diviser par 10 le coût des requêtes de Looker Studio.

Fonctionnalités clés :

  • Pipeline ELT complet & automatisé : Ingestion robuste de données horaires de prix d'électricité et exposition sous forme de table externe BigQuery.
  • Modélisation financière granulaire : Simulation de P&L (chiffre d'affaires requis, coûts totaux, coûts de l'énergie, profit brut) pour 10+ profils d'archetypes de PME.
  • Looker Studio Interactif : Dashboard dynamique avec filtres géographiques (zones de prix), sectoriels (type de PME) et temporels pour identifier l'exposition au risque.
  • Tests & Qualité des données : Utilisation rigoureuse de tests dbt (`not_null`, `unique`, seeds) assurant l'intégrité du pipeline avant publication.

GradeScale (Moteur d'évaluation IA & Infrastructure Cloud Azure)

Objectif : Concevoir une preuve de concept (PoC) d'un moteur d'évaluation formative backend assisté par IA permettant aux enseignants d'analyser de manière granulaire les copies d'élèves (par rubriques et critères, détection des misconceptions), avec une architecture transactionnelle robuste et une infrastructure cloud Azure automatisée.

Stack :

  • Logique Applicative : Node.js, TypeScript, Fastify, Zod (Structured Outputs & Validation), Prisma ORM, PostgreSQL
  • Cloud Infrastructure (Azure) : Azure Container Apps (ACA), Azure Static Web Apps (SWA), Azure PostgreSQL Flexible Server, Azure Key Vault
  • Intelligence Artificielle : Inférence Groq LPU (Modèles Llama 3.3 70B)
  • DevOps & Industrialisation : Terraform (IaC), Docker (Multi-stage), GitHub Actions (CI/CD), Vitest (QA)

Flux de Données & Évaluation IA

Workflow de traitement : Soumission de la copie -> Validation de schéma Zod -> Pseudonymisation -> Inférence structurée Groq LPU -> Persistance Prisma dans PostgreSQL Flexible Server.

Fonctionnalités clés :

  • Domain-Driven Design (DDD) & Modélisation Pédagogique : Structure hiérarchique dynamique représentant des compétences conceptuelles et analyse sémantique détectant les biais de raisonnement récurrents chez l'élève (misconceptions).
  • Privacy by Design (RGPD) : Couche de pseudonymisation interceptant et nettoyant les données étudiantes brutes avant leur envoi au fournisseur LLM externe.
  • Sécurité "Zero Secret" : Authentification sécurisée par Managed Identity (MSI) d'Azure connectant l'API Node.js directement à Azure Key Vault, évitant tout mot de passe en clair dans l'environnement.
  • CI/CD & DevOps complet : Pipeline automatisé (Vitest, Docker build sur GHCR, Terraform, Déploiement progressif sur ACA/SWA et tests de fumée).

Alpine Lambda Trade (Architecture Lambda Haute Performance)

Objectif : Concevoir une Architecture Lambda (Batch + Speed Layer) complexe traitant des flux financiers haute fréquence via websockets, avec un fort focus sur le FinOps (optimisation des coûts cloud) et la fiabilité des données (idempotence).

Stack :

  • Cloud & IaC : AWS (Lambda, S3, SQS), Terraform
  • Data Warehouse : Snowflake (Snowpipe automatique)
  • Transformations : dbt Core (Modèles incrémentaux, Tests de Qualité, Architecture Medallion)
  • Streaming : Producer Asynchrone Python (Binance WebSockets)
  • Serving : Streamlit

Architecture Haute Disponibilité

Flux Temps Réel (Speed) : Ingestion vers AWS SQS -> AWS Lambda (Calcul EMA/Anomalies en ms) -> Snowflake STG_REALTIME.
Flux Historique (Batch) : Bufferisation S3 (Parquet) -> Snowpipe -> dbt (Nettoyage & Déduplication) -> Snowflake SILVER.
Les deux couches fusionnent dynamiquement (UNION ALL) pour la Vue Métier finale sans duplication.
Architecture entièrement provisionnée par Terraform (IAM roles, SQS, Snowflake Integration).

Fonctionnalités clés & Impact Business :

  • Stratégie FinOps : Remplacement d'Amazon Kinesis par AWS SQS FIFO, réduisant l'empreinte cloud de 80% tout en garantissant l'ordre strict des messages et en éliminant les duplicatas.
  • Qualité & Fiabilité (dbt) : Pipelines idempotents avec modèles incrémentaux dbt. Utilisation massive de dbt test pour assurer les contraintes d'unicité et de non-nullité.
  • Détection d'anomalies : Calcul statistique (Bandes de Bollinger) on-the-fly par fonction Lambda Serverless (stateful warm cache).
  • Data Contracts : Gestion rigoureuse des erreurs asynchrones pour empêcher les corruptions de données (intégration possible de Dead Letter Queues - DLQ).

Market Data Pipeline Temps Réel (Kafka & SLI/SLO)

Objectif : Concevoir un Data Pipeline Temps Réel résilient de bout en bout pour des indices boursiers (SP500, STOXX600, NIKKEI225), de l’ingestion jusqu’au stockage, à l’API et aux tableaux de bord temps réel sous Linux.

Stack :

  • Streaming : Kafka (confluent-kafka), producteur/consommateur Python
  • Stockage : DuckDB
  • API : Flask (/live, /metrics)
  • Tableaux de bord : Grafana (datasource Infinity)
  • Runtime : Docker, Linux/WSL
  • CI : GitHub Actions (lint + tests de fumée)

Architecture de streaming

Inclut des métriques d'observabilité (SLI/SLO), des logs JSON structurés et des tests de fumée CI (GitHub Actions).

Fonctionnalités clés :

  • Streaming de bout en bout : Flux (mockés) ingérés dans Kafka avec architecture de résilience et persistés dans DuckDB utilisant des stratégies de Dead Letter.
  • API temps réel : Endpoints Flask exposant prix, volumes et agrégats en JSON.
  • Observabilité robuste : Métriques de run (messages traités, erreurs, durée, timestamp max) stockées dans DuckDB et loggées en JSON.
  • Tableaux de bord Grafana : Datasource Infinity interrogeant l’API pour afficher prix temps réel, volumes et tableaux de synthèse.
  • Pratiques DevOps : Kafka sous Docker, script bash runpipeline.sh et pipeline CI GitHub Actions pour valider le flux de bout en bout.

Suivi d’affluence en agence bancaire (Pipeline PySpark)

Objectif : Automatiser le suivi et l’analyse du trafic en agence bancaire afin de permettre l’optimisation de l’allocation des ressources (effectifs) en quasi temps réel.

Stack :

  • Ingestion / API : FastAPI
  • Traitement : PySpark
  • Orchestration : Airflow
  • Stockage : DuckDB (Serving OLAP) / Parquet Data Lake
  • Visualisation : Streamlit (application publique)
  • Conteneurisation : Docker

Workflow

Ingestion via FastAPI, lourds batchs PySpark distribués (exportés en parquet/iceberg), orchestrés par Airflow, et servis (layer OLAP) avec DuckDB in-process accelerateur et exposées aux analystes via un dashboard Streamlit.

Fonctionnalités clés :

  • Traitement à grande échelle : Jobs PySpark gérant des reprocess massifs (Backfills) et calculant des agrégats historisés (pics, moyennes, fenêtres glissantes) par agence et par plage horaire.
  • Intégration API : Endpoint FastAPI retournant les compteurs de visiteurs par agence pour les équipes opérationnelles.
  • Analyse de scénarios : Dashboard Streamlit permettant de comparer les agences, identifier les périodes de pointe et tester différents scénarios de staffing.
  • Workflows orchestrés : DAGs Airflow pour l’ingestion quasi temps réel et les recalculs historiques.
