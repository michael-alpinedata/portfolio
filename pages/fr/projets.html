1. Le Problème Business

Les dirigeants de PME subissent la volatilité des prix de gros de l'électricité sans aucune visibilité sur l'impact réel et immédiat sur leurs marges.

2. La Solution Apportée

Un dashboard analytique interactif permettant de simuler l'évolution de la marge brute selon le pays et le profil de l'entreprise, tout en optimisant strictement les coûts de la data (FinOps).

👉 Voir la solution Live (Dashboard)

3. Comment ça marche ?

Un pipeline automatisé extrait les données publiques du marché énergétique européen (OPSD), les croise avec des modèles comptables théoriques de PME, et consolide ces données en tables exploitables par la Business Intelligence.

🔗 Code source (Dépôt GitHub)

4. Justification des Choix Techniques

  • GCP & BigQuery : Le modèle de facturation "à la requête" (Serverless) est idéal et peu coûteux pour de l'analytique ponctuelle par rapport à une base allumée 24/7.
  • dbt Core : Permet de modulariser la logique financière complexe en SQL clair, de la tester (CI/CD), et de garantir la fiabilité des calculs.
  • Clustering & Partitionnement : Divise par 10 les coûts de requêtage du dashboard en limitant physiquement le volume de données scannées lors des filtrages utilisateurs.
  • Stack : GCP BigQuery dbt Core Terraform Looker Studio

1. Le Problème Business

La correction manuelle des copies empêche les enseignants de fournir un feedback personnalisé et granulaire par manque de temps. De plus, envoyer des copies d'élèves à des IA pose d'importants problèmes de confidentialité (RGPD).

2. La Solution Apportée

Un moteur d'évaluation Backend-as-a-Service qui analyse instantanément le raisonnement de l'élève de manière anonymisée, tout en renvoyant à l'enseignant un diagnostic structuré (méconceptions, notes, conseils).

👉 Tester la solution Live (Interface SWA)

3. Comment ça marche ?

L'API reçoit la copie et son barème, pseudonymise les informations personnelles, puis interroge de puissants modèles de langage (LLM) en forçant le format de la réponse, avant d'enregistrer le diagnostic dans une base sécurisée.

🔗 Code source (Dépôt GitHub)

4. Justification des Choix Techniques

  • Azure Container Apps : Une solution Serverless qui s'adapte automatiquement (scale) aux pics de charge imprévisibles liés aux périodes d'examens, tout en gardant des coûts maîtrisés.
  • Groq LPU (Llama 3.3) : Utilisation de processeurs spécialisés (LPU) permettant une inférence ultra-rapide et nettement moins coûteuse que les API propriétaires traditionnelles.
  • Zod & TypeScript : Strictement nécessaires pour valider le format de données renvoyé par l'IA (Structured Outputs), évitant que l'application ne crashe si l'IA hallucine ou change le format de sa réponse.
  • Stack : Node.js TypeScript Zod PostgreSQL Azure Cloud Groq LPU

1. Le Problème Business

En trading algorithmique haute fréquence, il faut réagir en moins d'une seconde aux pics de volatilité, mais également conserver un historique parfait et sans doublon pour l'entraînement des modèles. Maintenir ces deux besoins génère souvent des silos de données incohérents et des coûts cloud prohibitifs.

2. La Solution Apportée

Une infrastructure de données séparée en deux voies (Architecture Lambda) : une voie "Speed" pour les alertes instantanées et une voie "Batch" pour le stockage froid et l'auditabilité parfaite de l'historique.

3. Comment ça marche ?

Les transactions boursières sont captées en direct via WebSockets. Ce flux est scindé : l'un déclenche des analyses de volatilité immédiates via des fonctions Serverless, tandis que l'autre est stocké massivement "à froid" puis dédoublonné et consolidé.

🔗 Code source (Dépôt GitHub)

4. Justification des Choix Techniques

  • AWS SQS FIFO (au lieu de Kinesis) : Transition purement FinOps. SQS offre une facturation à l'usage et garantit l'ordre des messages financiers, réduisant la facture Cloud de 80% face à Kinesis qui nécessite une allocation continue.
  • AWS Lambda (Speed Layer) : Déclenchement instantané à chaque message reçu pour un calcul de volatilité ultra-rapide (Bandes de Bollinger) avec maintien de l'état en mémoire (Warm Start).
  • Snowflake & dbt (Batch Layer) : Snowflake est conçu pour le stockage analytique massif, tandis que dbt permet de garantir l'idempotence des données (purger les doublons causés par les micro-coupures réseau) via des modèles incrémentaux.
  • Stack : AWS (SQS, S3, Lambda) Python Async Snowflake dbt Core
